## היקף

### בתחום

- חשיפת מפתחות API או טוקנים בקוד
- ניהול sessions ו-cookies בלוח הבקרה
- הזרקת SQL דרך פרמטרים של ה-API
- SSRF (Server-Side Request Forgery) דרך `PROXY_URL`
- הרצת קוד שרירותי דרך קלט משתמש
- חשיפת מידע רגיש דרך הודעות שגיאה
- bypass של אימות לוח הבקרה

### מחוץ לתחום

- מתקפות DoS/DDoS
- בעיות אבטחה ב-dependencies של צד שלישי (דווח ישירות לספק)
- הגדרות שגויות של הסביבה על-ידי המשתמש
- חולשות הקשורות ל-API של פיקוד העורף עצמו

## שיקולי אבטחה ידועים

### לוח הבקרה

- **`DASHBOARD_SECRET`** — סיסמה יחידה להתחברות. מומלץ להשתמש בסיסמה חזקה (16+ תווים) ולא להחשיף את הפורט לאינטרנט הציבורי ישירות — שים Nginx עם TLS מלפנים.
- **Sessions** — token מאוחסן ב-Set פנימי בזיכרון. logout מבטל את ה-token מיידית. restart של השרת מבטל את כל הסשנים הפעילים.
- **Cookie** — `httpOnly`, `sameSite=strict`. בסביבת ייצור (HTTPS) מוגדר גם `secure=true`.
- **אין CSRF tokens** — ה-API מוגן על-ידי `sameSite=strict` בלבד.

### Telegram Bot Token

- **לעולם אל תכלול** את הטוקן ב-commit. `.env` נמצא ב-`.gitignore`.
- אם הטוקן נחשף — בטל אותו **מיידית** דרך [@BotFather](https://t.me/BotFather) וצור חדש.

### API פיקוד העורף

- ה-API ציבורי ולא דורש אימות — פגיעות בצד הספק אינן בשליטתנו.
- `PROXY_URL` לא עובר ולידציה — ודא שהפרוקסי אמין.

### SQLite

- מסד הנתונים מכיל chat IDs של משתמשים — הגן על הגישה לקובץ ה-DB.
- גיבויים (GET /api/settings/backup) מכילים את כל הנתונים — אבטח את הגישה לנקודת הקצה.
